# TravelSync 多人協作旅遊規畫系統：系統設計文件 (SDD) - MVP 版本

## 1. 簡介

### 1.1 專案概述
TravelSync 是一個旨在解決多人旅遊規劃痛點的協作平台。本系統透過集中化的即時編輯介面、直觀的時間軸視角以及民主的投票機制，消除通訊軟體中資訊碎片化的問題，讓旅遊規劃變得高效且具備高度參與感。

### 1.2 系統目標
*   **集中化管理**：提供單一真相來源（Single Source of Truth），所有成員看到的行程永遠是最新版本。
*   **即時協作體驗**：利用 Python 高效能框架模擬多人同時編輯的同步感。
*   **視覺化規劃**：將枯燥的文字列表轉化為具備時間感的垂直時間軸。
*   **簡化決策過程**：內建投票模組，讓「午餐吃什麼」等分歧能透過數據快速解決。
*   **低門檻加入**：透過連結快速加入行程，無需複雜的註冊流程。
*   **高擴展性架構**：採用 clean architecture 概念，便於未來整合 AI 建議或費用拆分功能。

### 1.3 技術選型
為了達成快速開發與本機易維護性，本系統採用以下技術棧：
*   **程式語言**：Python 3.10+ (強型別提示、易讀性高)
*   **Web UI 框架**：**Streamlit** (能快速將 Python 邏輯轉化為互動式網頁，適合 MVP 驗證)
*   **資料庫**：**SQLite** (單一檔案 `.db`，無需安裝配置資料庫伺服器)
*   **資料庫 ORM**：**SQLAlchemy** (簡化 SQL 操作，提供強型別支援)
*   **環境管理**：venv + pip

## 2. 系統架構與運作流程

### 2.1 整體架構
本系統採用單機全棧架構，前端 UI 與後端邏輯高度整合於 Streamlit 運行環境中。

```text
[使用者瀏覽器] <--(HTTPS/WebSocket)--> [Streamlit App (app.py)]
                                            |
                                    [核心業務模組 (travel_logic.py)]
                                            |
                                    [資料持久層 (database.py)]
                                            |
                                      [SQLite 資料庫檔案]
```

### 2.2 運作流程詳解
1.  **初始化/登入**：使用者啟動 App，輸入暱稱並選擇「建立新行程」或「透過連結加入」。
2.  **行程管理**：主辦人設定旅遊日期與名稱，系統在 SQLite 中建立紀錄。
3.  **協作規劃**：
    *   成員點選「新增景點」，系統調用 Google Places API (或模擬搜尋) 獲取資料。
    *   資料寫入資料庫後，Streamlit 觸發所有在線用戶的頁面刷新（Rerender），達成即時同步效果。
4.  **投票決策**：針對爭議時段發起提案，成員點擊按鈕投票，結果即時統計並顯示在介面上。
5.  **行程瀏覽**：系統將資料庫中的活動按時間排序，渲染成時間軸介面供用戶查閱。

## 3. 核心模組設計

本系統將功能拆解為以下核心模組：

### 3.1 `models.py` (資料模型模組)
*   **職責**：定義系統的所有實體物件與資料庫對應關係。
*   **核心功能**：使用 SQLAlchemy 定義 `User`, `Trip`, `ScheduleItem`, `Vote` 等類別。

### 3.2 `database.py` (資料庫連結模組)
*   **職責**：處理資料庫初始化、連線池管理及基礎 CRUD (增刪查改) 操作。
*   **核心功能**：
    *   `init_db()`: 建立資料表。
    *   `get_db_session()`: 獲取資料庫連線。

### 3.3 `travel_logic.py` (業務邏輯模組)
*   **職責**：處理旅遊規劃的具體運算邏輯。
*   **核心功能**：
    *   `add_activity()`: 驗證時間重疊並新增景點。
    *   `calculate_votes()`: 統計各選項票數並回傳獲勝者。
    *   `reorder_timeline()`: 處理拖放後的排序邏輯。

### 3.4 `app.py` (介面與控制器模組)
*   **職責**：定義 Streamlit UI 佈局，捕捉使用者行為並呼叫邏輯模組。
*   **核心功能**：
    *   `render_sidebar()`: 顯示成員狀態與行程列表。
    *   `render_timeline()`: 渲染每日行程卡片。
    *   `render_voting_section()`: 顯示投票介面。

## 4. 資料庫設計

### 4.1 資料庫選型：SQLite
*   **原因**：
    1.  **零配置**：開發者只需安裝 Python 套件即可運行。
    2.  **可攜性**：整個資料庫是一個檔案，方便開發團隊共享測試數據。
    3.  **效能足夠**：對於 10 人以下的協作，SQLite 的併發處理能力綽綽有餘。

### 4.2 資料表設計

#### 4.2.1 `trips` (行程表)
| 欄位名稱 | 資料型態 | 說明 | 備註 |

| title | VARCHAR | 旅遊標題 | |
| start_date | DATE | 開始日期 | |

| invite_code | VARCHAR | 邀請連結碼 | 隨機生成的唯一字串 |

| 欄位名稱 | 資料型態 | 說明 | 備註 |

## 5. 使用者介面與互動規劃

### 5.1 頁面結構
*   **頂部導覽欄**：顯示行程名稱、目前在線成員頭像（模擬）。
*   **左側側邊欄**：
    *   行程切換器。
    *   邀請連結複製按鈕。
    *   目前的權限狀態（主辦人/編輯者）。
*   **主工作區**：
    *   **日期切換標籤**：Day 1, Day 2...
    *   **時間軸區域**：由上而下排列的景點卡片。
    *   **懸浮動作按鈕 (FAB)**：新增景點或發起投票。

### 5.2 核心互動流程
1.  **新增景點**：點擊「+」 -> 彈出視窗輸入名稱或搜尋 -> 選擇時間 -> 資料即時顯示在時間軸。
2.  **發起投票**：選擇「建立提案」 -> 輸入多個地點 -> 其他成員在各自畫面上會看到「投票中」的黃色區塊 -> 點擊按鈕投票。
3.  **調整順序**：點擊卡片上的「上移/下移」箭頭，介面立即重新排序。

## 6. API 設計 / 功能函數

### 6.1 `add_activity(trip_id, data)`
*   **輸入**：行程 ID, 包含名稱、時間、地點的字典。
*   **輸出**：成功或失敗的狀態。
*   **職責**：將新活動寫入 `schedule_items` 表，並將其 `is_proposal` 設為 False。

### 6.2 `create_proposal(trip_id, options)`
*   **輸入**：行程 ID, 多個地點選項。
*   **輸出**：產生的提案群組 ID。
*   **職責**：在資料庫建立多筆 `is_proposal=True` 的紀錄。

### 6.3 `finalize_vote(proposal_id)`
*   **輸入**：提案 ID。
*   **職責**：統計 `votes` 表中票數最高者，將該項目的 `is_proposal` 改為 False，其餘選項刪除。

## 7. 錯誤處理策略

*   **網路中斷或 API 失敗**
    *   **處理策略**：使用 Python 的 `try-except` 封裝 Google API 調用。
    *   **UI 呈現**：在頁面頂端彈出 `st.error("無法取得地圖資訊，請手動輸入地址")`。
*   **資料併發衝突 (同時修改同一景點)**
    *   **處理策略**：採用「最後寫入者勝」原則，但在修改前檢查 `updated_at` 時間戳記。
    *   **UI 呈現**：若偵測到衝突，顯示警告訊息並請用戶重新載入。
*   **無效的邀請連結**
    *   **處理策略**：查詢資料庫若無對應 `invite_code` 則導向 404 頁面。
    *   **UI 呈現**：顯示「此行程不存在或已被刪除」。

## 8. 實作路徑 (Implementation Roadmap)

### 8.1 環境建置與依賴安裝
```bash
# 建立專案資料夾
mkdir travelsync_mvp
cd travelsync_mvp

# 建立虛擬環境
python -m venv venv
source venv/bin/activate  # Mac/Linux
# .\venv\Scripts\activate # Windows
```

**requirements.txt** 內容：
```text
streamlit==1.32.0
sqlalchemy==2.0.25
pandas
requests
python-dotenv
```

安裝指令：
```bash
pip install -r requirements.txt
```

**.gitignore** 內容：
```text
venv/
*.db
.env
__pycache__/
```

### 8.2 資料庫模組開發 (`database.py`)
1.  實作 SQLAlchemy 的 `Base` 模型與 `engine` 配置。
2.  編寫 `create_db_and_tables()` 函數，在程式啟動時確保 SQLite 檔案與資料表已建立。
3.  實作 Session 工廠，確保每次存取後能正確關閉連線。

### 8.3 核心業務邏輯開發 (`travel_logic.py`)
1.  撰寫 `get_trip_data(trip_id)`：從資料庫抓取所有定案行程與提案。
2.  撰寫 `handle_voting(user_id, item_id)`：處理投票邏輯，防止重複投票。
3.  實作 Google Places 模擬器：若無 API Key，先回傳假資料以利開發進度。

### 8.4 使用者介面開發 (`app.py`)
1.  **第一步：基礎佈局**
    - 使用 `st.set_page_config` 設定頁面標題。
    - 使用 `st.sidebar` 實作行程導覽。
2.  **第二步：狀態管理**
    - 使用 `st.session_state` 儲存目前的行程 ID 與用戶資訊。
3.  **第三步：元件開發**
    - 使用 `st.tabs` 實作每日行程切換。
    - 使用 `st.container` 與 `st.columns` 渲染行程卡片。
    - 整合 `st.form` 用於新增景點的表單。

### 8.5 測試與驗證
*   **冒煙測試**：建立一個行程 -> 新增 3 個景點 -> 檢查資料庫檔案是否生成。
*   **協作測試**：開啟兩個瀏覽器視窗（一個無痕模式），在 A 視窗修改，檢查 B 視窗刷新後是否出現更動。
*   **邊界測試**：輸入極端的旅遊日期（如 30 天）檢查 UI 是否崩潰。

### 8.6 部署與運行說明
在本機環境下，執行以下指令啟動系統：
```bash
streamlit run app.py
```
系統將自動開啟瀏覽器視窗，預設網址為 `http://localhost:8501`。
