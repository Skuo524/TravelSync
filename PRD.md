# 產品需求文件 (PRD)：TravelSync 多人協作旅遊規畫系統

---

## 1. 簡介

### 1.1 文件目的
本文件旨在詳述「TravelSync 多人協作旅遊規畫系統」之最小可行產品 (MVP) 的功能需求、技術架構、用戶流程及成功指標。本文件將作為開發團隊（前端、後端、UI/UX 設計師）與專案關係人之間的正式溝通基準，確保所有參與者對產品範圍與執行細節達成共識。

### 1.2 產品背景
旅遊規劃通常涉及大量的資訊收集與決策過程。當場景轉移到「多人出遊」時，痛點會呈指數級增長：成員間的偏好不一、通訊軟體（如 LINE, WhatsApp）中的資訊碎片化、難以達成共識的投票過程，以及缺乏一個能讓所有人即時看到最新版本行程的工具。目前的解決方案（如 Google Sheets 或單機版 App）往往缺乏即時互動性或地圖整合能力。

### 1.3 產品願景
TravelSync 旨在成為全球領先的協作旅遊平台，讓「規劃行程」本身成為旅遊樂趣的一部分。我們透過直觀的即時同步技術與民主化的決策工具，消除群體旅遊規劃中的摩擦，讓每位成員都能輕鬆貢獻想法，最終產出完美的旅遊時間表。

### 1.4 MVP 目標
*   **建立核心協作引擎**：實現類似 Google Docs 的即時編輯體驗，確保多人在線編輯時數據不衝突。
*   **視覺化時間軸規劃**：提供直觀的拖放式時間軸，讓行程安排具備高度結構化與易讀性。
*   **簡化資訊獲取**：整合 Google Maps API，讓用戶無需手動輸入地址，能快速獲取景點資訊。
*   **建立民主決策機制**：內建針對特定時段的投票系統，解決多人意見分歧。
*   **快速加入與權限控管**：透過簡單的連結分享，實現快速加入行程，並區分管理者與參與者權限。
*   **跨裝置瀏覽最佳化**：確保網頁版在桌機與手機瀏覽器上皆能流暢操作。

## 2. 目標用戶

### 主要用戶：多人協作族群
*   **朋友圈規劃者**：負責發起活動，但希望減輕溝通成本、收集朋友意見的領頭者。
*   **家庭旅遊發起人**：需要協調長輩偏好與小孩需求的父母。
*   **學生畢業旅行團**：預算有限、偏好明確且需要高度民主討論的學生群體。

### 用戶特徵
*   **技術能力**：中等，熟悉一般的網頁操作與 Google Maps。
*   **需求痛點**：討厭在通訊軟體中翻找訊息、對於決定要去哪裡感到疲憊、希望行程表能一目了然。
*   **使用行為**：通常在桌機進行複雜的大範圍規劃，在旅途中則使用手機瀏覽行程。

## 3. 核心問題與解決方案

*   **核心問題 1：資訊分散且同步延遲**
    *   *問題描述*：成員在不同的 LINE 群組發送景點連結，最後沒人知道最終版本是什麼。
    *   **MVP 解決方案**：實時同步編輯模式。所有成員在同一畫面上操作，任何更動（新增景點、修改時間）都會在 500ms 內同步給所有在線用戶。

*   **核心問題 2：意見分歧難以達成共識**
    *   *問題描述*：兩個人想去不同的餐廳，在群組裡討論半天沒有結論。
    *   **MVP 解決方案**：提案與投票機制。針對特定時段建立「候選清單」，成員可針對選項點擊「贊成」，系統自動統計票數。

*   **核心問題 3：規劃過程耗時且繁瑣**
    *   *問題描述*：需要手動複製景點地址、營業時間，還要預估交通時間。
    *   **MVP 解決方案**：半自動資訊匯入。搜尋景點後自動抓取 Google 地圖資訊，並提供基礎框架讓用戶手動設定交通時間。

*   **核心問題 4：缺乏視覺化的時程感**
    *   *問題描述*：列表式的行程很難判斷景點與景點之間是否太趕或太遠。
    *   **MVP 解決方案**：時間軸模式（Timeline View）。以垂直或水平時間軸呈現，讓行程的「空檔」與「重疊」一目了然。

## 4. 產品功能 (MVP 範圍)

### 4.1 行程管理中心
*   **功能描述**：用戶管理所有過去、現在與未來的旅遊計畫。
*   **具體子功能**：
    *   **建立新行程**：輸入旅遊名稱、選擇日期區間（起訖日）、上傳封面縮圖。
    *   **行程列表看板**：顯示用戶參與的所有行程，並標註「我是主辦人」或「我是參與者」。
    *   **行程複製與刪除**：允許主辦人刪除行程，或讓用戶複製現有行程作為範本。

### 4.2 實時協作與權限系統
*   **功能描述**：確保多人同時在線的協作邏輯。
*   **具體子功能**：
    *   **邀請連結生成**：產生唯一加密連結，收到連結的人登入後即可加入該行程。
    *   **角色區分**：分為「主辦人（Admin）」與「編輯者（Editor）」。主辦人可刪除整個行程或管理成員；編輯者可操作時間軸。
    *   **在線成員狀態**：在頂端顯示目前有哪些成員正在線上查看或編輯。
    *   **編輯鎖定機制**：當某人正在編輯特定景點細節時，其他用戶會看到該區塊被鎖定，防止數據覆蓋。

### 4.3 時間軸規劃器 (Core)
*   **功能描述**：產品的核心介面，用於排定每日行程。
*   **具體子功能**：
    *   **每日分頁導覽**：以日期標籤切換不同的旅遊天數。
    *   **拖放式排序 (Drag & Drop)**：用戶可自由上下拖動景點區塊以更換先後順序。
    *   **時間塊設定**：設定每個景點的開始時間與預計停留時長。
    *   **交通資訊欄位**：在兩個景點區塊之間，提供手動輸入交通方式（步行、開車、大眾運輸）與交通時間的欄位。

### 4.4 景點搜尋與匯入 (Google Maps 整合)
*   **功能描述**：簡化景點資料輸入流程。
*   **具體子功能**：
    *   **關鍵字搜尋景點**：整合 Google Places API，支持名稱、地址搜尋。
    *   **自動帶入基本資訊**：選擇景點後，自動填入地址、照片、營業時間、官方網站連結。
    *   **自定義地點**：允許用戶手動輸入非地圖上的地點（如：某人的家）。

### 4.5 提案與投票模組
*   **功能描述**：解決決策衝突的工具。
*   **具體子功能**：
    *   **發起提案**：在特定時段（如午餐時間）建立「提案槽」，並加入多個候選景點。
    *   **匿名/記名投票**：成員可對心儀的候選景點投下贊成票。
    *   **決策轉化**：主辦人可根據投票結果，一鍵將獲勝提案轉化為正式行程。

### 4.6 景點細節與留言討論
*   **功能描述**：針對特定景點的細節溝通。
*   **具體子功能**：
    *   **備註欄位**：記錄該景點的注意事項（如：訂位編號、必點菜色）。
    *   **留言區**：每個行程項目下設有獨立留言板，供成員討論細節，不與主行程混亂。
    *   **附件上傳**：允許上傳圖片（如門票截圖、預約確認信）。

## 5. 用戶故事 (User Stories)

1.  **作為** 主辦人，**我想要** 建立一個為期五天的日本行程，**以便** 我能開始邀請朋友們一起規劃。
2.  **作為** 編輯者，**我想要** 在搜尋框輸入「東京鐵塔」後直接將它加入時間軸，**以便** 我不需要手動複製地址。
3.  **作為** 成員，**我想要** 在時間軸上看到其他人正在操作哪個景點，**以便** 我們不會同時修改同一個內容導致錯誤。
4.  **作為** 成員，**我想要** 針對週六晚上的餐廳發起投票，**以便** 決定大家最想吃哪一家燒肉。
5.  **作為** 規劃者，**我想要** 隨意拖動景點的順序，**以便** 根據地理位置優化我們的動線。
6.  **作為** 用戶，**我想要** 透過點擊連結就加入行程，**以便** 略過繁瑣的搜尋與申請流程。
7.  **作為** 主辦人，**我想要** 在景點備註欄填寫預約時間，**以便** 所有成員都能隨時查看。
8.  **作為** 用戶，**我想要** 在手機上查看已定案的行程，**以便** 我在旅途中能隨時確認下一個目的地。

## 6. 非 MVP 範圍 (未來考量)

*   **AI 智能規劃**：根據用戶偏好自動生成初始行程草稿。
*   **自動路徑優化**：利用演算法自動排列景點順序，使交通時間最短化。
*   **費用拆分系統 (Split-wise 整合)**：記錄旅遊開支並自動計算每人應分擔金額。
*   **離線模式**：在無網路環境下（如飛機上或山區）仍可瀏覽行程。
*   **原生 App (iOS/Android)**：提供更流暢的行動端體驗與推播通知。
*   **整合航空公司/飯店 API**：自動匯入機票與住宿確認資訊。
*   **社交分享功能**：允許用戶公開分享其行程範本供他人參考。
*   **氣象預報整合**：在時間軸上顯示旅遊地的即時天氣預測。
*   **PDF/地圖匯出**：將行程匯出成精美的 PDF 或匯入至 Google My Maps。
*   **多語言支援**：初期僅支援繁體中文，未來擴及英文、日文。

## 7. 技術要求

### 7.1 後端框架
*   **推薦**：**Node.js (NestJS 框架)**
*   **理由**：Node.js 的非同步 I/O 特性非常適合處理高併發的 WebSocket 連接（用於實時同步）。NestJS 提供結構嚴謹的架構，有利於未來功能擴充與維護。

### 7.2 資料庫
*   **推薦**：**PostgreSQL + Redis**
*   **理由**：PostgreSQL 適合儲存結構化的行程與用戶數據（關係明確）；Redis 用於快取即時編輯狀態，降低資料庫負擔並提升同步速度。

### 7.3 資料庫 ORM
*   **推薦**：**Prisma**
*   **理由**：Prisma 提供強型別的支援與極佳的開發者體驗，能自動生成 API 層的型別定義，減少開發錯誤。

### 7.4 前端技術
*   **推薦**：**React (Next.js)** + **Tailwind CSS** + **Zustand (狀態管理)**
*   **理由**：Next.js 支持 SSR/SSG，有利於未來若需做 SEO；Tailwind CSS 能快速建構響應式介面；Zustand 輕量且適合處理複雜的編輯狀態。
*   **實時技術**：**Socket.io**。用於實現多用戶間的雙向實時通信。

### 7.5 其他整合
*   **Google Places API**：用於景點搜尋與資訊自動帶入。
*   **AWS S3**：用於存儲用戶上傳的封面圖或附件。
*   **Firebase Authentication**：快速實現社交登入（Google/Facebook）。

### 7.6 部署環境
*   **前端**：Vercel（與 Next.js 整合度最高）。
*   **後端**：AWS EC2 或 Docker 容器化部署於 Google Cloud Run。

### 7.7 版本控制
*   使用 **Git** 並託管於 **GitHub**。建議採用 Git Flow 工作流進行版本管理。

## 8. 成功指標

*   **規劃效率提升**：用戶在系統上完成一個 3 天行程所需的時間，較傳統通訊軟體協作減少 40% 以上。
*   **行程完成率**：超過 60% 被建立的行程最終能產出至少 5 個以上的定案景點（代表系統真的被用於規劃而非測試）。
*   **協作參與度**：平均每個行程的參與人數達到 3 人以上。
*   **系統同步延遲**：多人編輯時的數據同步延遲需控制在 500ms 以內。
*   **用戶留存率**：完成一次旅遊規劃後，用戶在下次旅遊時再次使用系統的比例。

## 9. 假設與約束

### 9.1 假設
*   用戶在規劃行程時擁有穩定的網路連接（因為 MVP 暫無離線功能）。
*   Google Maps API 的收費標準在專案預算可負擔範圍內。
*   用戶願意使用社交帳號（Google）進行登入以獲取最佳體驗。
*   多數用戶的主要規劃場景發生在桌上型電腦或平板電腦。
*   行程參與人數在 10 人以內（超過 10 人對即時同步技術與 UI 呈現將是極大挑戰）。

### 9.2 約束
*   **開發週期**：首個 MVP 版本需在 3 個月內開發完成。
*   **成本約束**：必須優化 Google API 的調用頻次，避免產生高額費用。
*   **平台限制**：首版僅支援 Web 端，不開發原生行動 App，但必須保證手機瀏覽器（Mobile Safari/Chrome）的可用性。
*   **法規約束**：必須符合 GDPR 或相關隱私法規，妥善保護用戶的旅遊習慣與位置數據。
